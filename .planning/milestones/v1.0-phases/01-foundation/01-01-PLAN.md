<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Filesystem → reader | Path passed to read_resume comes from config.py constant (hardcoded), not user input in Phase 1 |
| Writer → filesystem | content passed to write_resume comes from the LLM in Phase 2; in Phase 1 it is caller-provided |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | resume_reader.py — file path | accept | BASE_RESUME_PATH is a compile-time constant in config.py, not user-controlled in Phase 1. No path traversal vector exists. Phase 3 may add --resume flag; that phase must validate the path. |
| T-01-02 | Tampering | resume_writer.py — output directory | accept | OUTPUT_DIR is anchored to Path(__file__) in config.py. No user-supplied path in Phase 1. Writer cannot be redirected by cwd manipulation. |
| T-01-03 | Tampering | resume_writer.py — file content | accept | Phase 1 content is never user-supplied. Phase 2 (llm_client.py) owns content validation before write_resume is called; that phase will strip fences and validate LaTeX structure. |
| T-01-04 | Information Disclosure | resume_reader.py — stderr message | accept | Error message prints the file path to stderr. This is a local CLI; no remote attacker can observe stderr. Path disclosure to the local user is intentional (helps them fix the config). |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | Phase 1 installs no new packages via package manager commands (all files are created directly). pyproject.toml declares hatchling as build backend — approved in Package Legitimacy Audit in RESEARCH.md. |
</threat_model>
